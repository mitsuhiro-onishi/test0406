# 要件定義書

# 展示会ドキュメント一元管理システム

---

## 1. プロジェクト背景

### 1.1 現状の課題

展示会の運営において、主催者および装飾業者（全体管理委託先）は、多数の出展社から注文書・設計書等のドキュメントを受領し管理する必要がある。現状では以下の問題が発生している。

| # | 課題 | 影響 |
|---|------|------|
| 1 | ドキュメント形式がバラバラ（Excel, Word, PDF, 手書きFAX等） | 確認・転記に膨大な時間がかかる |
| 2 | 送信方法が統一されていない（メール, FAX, 手渡し） | 受領漏れ、紛失リスク |
| 3 | 手作業での集計・管理 | 転記ミス、集計ミスが発生 |
| 4 | 提出状況の把握が困難 | 締切間際に未提出が判明し混乱 |
| 5 | 過去ドキュメントの検索が困難 | 前回実績の参照に時間がかかる |

### 1.2 なぜ既存ツールでは解決できないか

- **出展社に専用ツールの利用を強制できない**: 出展社は自社のフォーマット・業務フローを持っている。主催者側の都合で新ツールへの入力を求めると反発が起きる
- **汎用のファイル共有ツール（Google Drive等）では不十分**: ファイルは集まるが、内容の構造化・集計・進捗管理ができない
- **既存の展示会管理システム**: 書類の内容解析まではカバーしていない

### 1.3 ソリューションの方向性

> 出展社の業務フローは変えない。AIが「形式の違い」を吸収し、主催者には統一された構造化データとして提供する。

---

## 2. システムの目的とゴール

### 2.1 目的

展示会運営に関わるドキュメント（注文書・設計書等）を、**形式や送信手段を問わず受け入れ**、AIによる自動解析で**構造化データとして一元管理**するシステムを構築する。

### 2.2 達成すべきゴール

| # | ゴール | 定量目標 |
|---|--------|---------|
| G-1 | ドキュメント受領の一元化 | 全チャネル（Web/メール/FAX）を1システムで受け入れ |
| G-2 | AI自動解析による工数削減 | 手作業での転記作業を80%以上削減 |
| G-3 | リアルタイム進捗把握 | 提出状況をダッシュボードで即座に確認可能 |
| G-4 | 出展社の負担ゼロ | 出展社は既存のドキュメントをそのまま送るだけ |
| G-5 | 正確なデータ管理 | AI解析＋人間レビューで転記ミスをゼロに |

## 3. ステークホルダー

### 3.1 ユーザー区分

```
┌──────────────────────────────────────────────────────────┐
│ 主催者（イベントオーナー）                                 │
│  例: 展示会主催会社、見本市協会                            │
│  ・全体の統括管理                                         │
│  ・最終的な意思決定権限を持つ                              │
├──────────────────────────────────────────────────────────┤
│ 装飾業者（全体管理委託先）                                 │
│  例: 展示装飾会社                                         │
│  ・主催者から全体管理を委託されている                       │
│  ・出展社からの書類を直接受け取る実務担当                   │
│  ・施工の計画・実行                                        │
├──────────────────────────────────────────────────────────┤
│ 出展社（各ブース出展企業）                                 │
│  例: メーカー、代理店、サービス企業                        │
│  ・ブースの注文書・設計書を提出する                         │
│  ・数十社〜数百社と規模は展示会による                       │
│  ・ITリテラシーはまちまち                                  │
└──────────────────────────────────────────────────────────┘
```

### 3.2 関連する外部システム・サービス

| システム | 関連 |
|---------|------|
| メールサーバー | 出展社からのメール受信 |
| FAXサービス | 出展社からのFAX受信 |
| AI/LLMサービス | ドキュメント解析 |
| OCRサービス | 画像・スキャンPDFのテキスト化 |
| ファイルストレージ | 原本ファイルの保管 |

## 6. 業務フロー

### 6.1 メインフロー：ドキュメント受信〜データ確定

```
出展社                    システム                    装飾業者/主催者
  │                         │                            │
  │  ① 書類送付             │                            │
  │  (Web/メール/FAX)       │                            │
  │ ───────────────────►    │                            │
  │                         │                            │
  │                    ② ファイル受信                     │
  │                    ③ ウイルススキャン                  │
  │                    ④ 原本保管                        │
  │                    ⑤ テキスト抽出                     │
  │                    ⑥ AI構造化解析                     │
  │                    ⑦ 信頼度スコア算出                 │
  │                         │                            │
  │                         ├── 信頼度 ≥ 閾値 ──►  自動確定
  │                         │                     ダッシュボードに反映
  │                         │                            │
  │                         ├── 信頼度 < 閾値 ──►  ⑧ レビュー依頼通知
  │                         │                            │
  │                         │                    ⑨ レビュー画面で確認
  │                         │                       原本と解析結果を
  │                         │                       並べて表示
  │                         │                            │
  │                         │                    ⑩ 承認/修正/却下
  │                         │  ◄───────────────────────  │
  │                         │                            │
  │                    ⑪ 確定データをDB格納               │
  │                    ⑫ ダッシュボード更新               │
  │                         │                            │
```

### 6.2 サブフロー：メール受信の自動処理

```
出展社が通常のメールで書類を送付
  │
  ▼
専用メールアドレスで受信（例: docs-ex15@example.com）
  │
  ▼
送信元メールアドレスから出展社を自動特定
  │
  ├── 特定成功 → 添付ファイルを自動登録、AI解析開始
  │
  └── 特定失敗 → 装飾業者に通知「不明な送信元からの書類受信」
                  手動で出展社を紐付け
```

### 6.3 サブフロー：FAX受信の自動処理

```
出展社がFAXで書類を送付
  │
  ▼
FAXサービスがTIFF/PDF画像として受信
  │
  ▼
Webhook経由でシステムに通知
  │
  ▼
OCR処理でテキスト化
  │
  ▼
LLM解析（※FAXは画質が低いため信頼度が低くなりやすい）
  │
  ▼
多くの場合レビューキューに入る → 人間が確認
```

## 7. 制約条件

| # | 制約 | 理由 |
|---|------|------|
| 1 | 出展社に専用ツールの利用を強制しない | ビジネス上の制約。出展社との関係性 |
| 2 | 日本語のドキュメントが主対象 | 国内展示会が対象 |
| 3 | AI解析は100%の精度を保証しない | 人間レビューとの併用で品質担保 |
| 4 | 個人情報の取り扱いに注意 | 出展社担当者の氏名・連絡先等が含まれる |

## 9. 成功基準

| # | 指標 | 目標値 | 測定方法 |
|---|------|--------|---------|
| 1 | ドキュメント転記作業時間の削減率 | 80%以上 | 導入前後の作業時間比較 |
| 2 | AI解析の自動確定率（人間レビュー不要） | 70%以上 | 信頼度閾値超えの割合 |
| 3 | 出展社の書類提出率（締切時点） | 95%以上 | 提出済み/全出展社 |
| 4 | 転記ミス発生率 | 0.5%以下 | エラー件数/全データ件数 |
| 5 | 出展社のシステム満足度 | 4.0/5.0以上 | アンケート調査 |

## 10. リスクと対策

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| AI解析精度が想定を下回る | 高 | 中 | プロンプトチューニング、学習データの蓄積、信頼度閾値の調整 |
| 出展社がポータルを使わない | 中 | 高 | メール/FAXでの受付を必ず維持。ポータルは「便利なオプション」として提供 |
| FAXの画質が低く読み取れない | 中 | 中 | 読み取り不可の場合は出展社に再送依頼の自動通知 |
| AI APIの料金が想定以上 | 中 | 低 | コスト監視、キャッシュ活用、バッチ処理による効率化 |
| 個人情報漏洩 | 高 | 低 | 暗号化、アクセス制御、監査ログ、定期セキュリティ監査 |

## 11. 用語定義

| 用語 | 定義 |
|------|------|
| 主催者 | 展示会を主催する企業・団体 |
| 装飾業者 | 主催者から展示会の全体管理（ブース施工含む）を委託された企業 |
| 出展社 | 展示会にブースを出展する企業 |
| 注文書 | 出展社がブース設備（パネル、照明、電気工事等）を注文する書類 |
| 設計書 | ブースの設計図面・仕様書 |
| AI解析 | LLM（大規模言語モデル）によるドキュメントの構造化データ抽出 |
| 信頼度スコア | AI解析結果の確からしさを示す数値（0.00〜1.00） |
| レビューキュー | 人間の確認が必要なAI解析結果の待ち行列 |
