# 要件定義書

# 展示会ドキュメント一元管理システム

---

## 1. プロジェクト背景

### 1.1 現状の課題

展示会の運営において、主催者側（主催者・装飾業者・電気施工会社等）は、
多数の出展社から注文書・設計書等のドキュメントを受領し管理する必要がある。
現状では以下の問題が発生している。

| # | 課題 | 影響 |
|---|------|------|
| 1 | ドキュメント形式がバラバラ（Excel, Word, PDF等） | 確認・転記に膨大な時間がかかる |
| 2 | 送信方法が統一されていない（メール, 手渡し等） | 受領漏れ、紛失リスク |
| 3 | 手作業での集計・管理 | 転記ミス、集計ミスが発生 |
| 4 | 提出状況の把握が困難 | 締切間際に未提出が判明し混乱 |
| 5 | 過去ドキュメントの検索が困難 | 前回実績の参照に時間がかかる |

### 1.2 なぜ既存ツールでは解決できないか

- **出展社に専用ツールの利用を強制できない**:
  出展社は自社のフォーマット・業務フローを持っている。
  主催者側の都合で新ツールへの入力を求めると反発が起きる

- **汎用のファイル共有ツール（Google Drive等）では不十分**:
  ファイルは集まるが、
  内容の構造化・集計・進捗管理ができない

- **既存の展示会管理システム**:
  書類の内容解析まではカバーしていない

### 1.3 ソリューションの方向性

> 出展社の業務フローは変えない。
> AIが「形式の違い」を吸収し、
> 主催者側には統一された構造化データとして提供する。

---

## 2. システムの目的とゴール

### 2.1 目的

展示会運営に関わるドキュメント（注文書・設計書等）を、
**形式を問わずWebブラウザからアップロードで受け入れ**、
AIによる自動解析で
**構造化データとして一元管理**するシステムを構築する。

### 2.2 達成すべきゴール

| # | ゴール | 定量目標 |
|---|--------|---------|
| G-1 | ドキュメント受領の一元化 | Webアップロードで1システムに集約 |
| G-2 | AI自動解析による工数削減 | 手作業での転記作業を80%以上削減 |
| G-3 | リアルタイム進捗把握 | 提出状況をダッシュボードで即座に確認可能 |
| G-4 | 出展社の負担最小化 | 既存ドキュメントをそのままアップロードするだけ |
| G-5 | 正確なデータ管理 | AI解析＋人間レビューで転記ミスをゼロに |

## 3. ステークホルダー

### 3.1 送受信の関係整理

```
【送る側（出展社）】               【受け取る側（主催者側）】

  出展社は複数の書類を              主催者
  それぞれの宛先に送る                │
                                     ├── 装飾業者 ← コマ申込、ブース設営
  出展社A ─┬─ コマ申込書             ├── 電気施工会社 ← 電気申込
  　　　　 ├─ 電気申込書             ├── ケータリング会社 ← 弁当注文
  　　　　 ├─ 弁当注文書             └── 備品レンタル会社 ← 備品申込
  　　　　 ├─ 備品申込書
  　　　　 └─ ブース設計書      ※ どの提出カテゴリを誰が受け取るかは
                                  展示会ごとに異なる
```

### 3.2 提出カテゴリと受取先の関係（展示会ごとに設定）

展示会ごとに「出展社が何を提出するか（提出カテゴリ）」と
「誰がそれを受け取るか（受取先組織）」のセットを自由に設定できる。

```
【展示会A の設定例】              【展示会B の設定例】
┌──────────┬──────────────┐    ┌──────────┬──────────────┐
│提出カテゴリ│受取先         │    │提出カテゴリ│受取先         │
├──────────┼──────────────┤    ├──────────┼──────────────┤
│コマ申込   │ 装飾業者A     │    │ブース設営 │ 装飾業者X     │
│ブース設営 │ 装飾業者A     │    │電気申込   │ 電気会社Y     │
│電気申込   │ 電気施工会社B │    │備品レンタル│ 装飾業者X     │
│弁当注文   │ ケータリングC │    │清掃依頼   │ 清掃会社Z     │
│備品レンタル│ 装飾業者A     │    └──────────┴──────────────┘
└──────────┴──────────────┘

※ 提出カテゴリの種類・数も展示会ごとに自由
※ 1つの受取先が複数カテゴリを担当可能
※ 主催者は全カテゴリ・全受取先を横断して管理可能
```

### 3.3 ユーザー区分

**主催者（イベントオーナー）**
- 例: 展示会主催会社、見本市協会
- 全体の統括管理
- 最終的な意思決定権限を持つ

**装飾業者（全体管理委託先）**
- 主催者から全体管理を委託されている
- 出展社からの書類を直接受け取る実務担当
- ブース施工の計画・実行

**主催者側の協力会社**
- 電気施工会社: 電気工事関連の注文を受ける
- 弁当・ケータリング会社: 飲食関連の発注を受ける
- その他: 展示会運営に関わる各種業者
- これらも主催者側として書類を受け取る立場

**出展社（各ブース出展企業）**
- 例: メーカー、代理店、サービス企業
- ブースの注文書・設計書を提出する側
- 数十社〜数百社と規模は展示会による
- ITリテラシーはまちまち
- 現在は申込要件に沿って各社宛に個別送付している

### 3.4 関連する外部システム・サービス

| システム | 関連 |
|---------|------|
| AI/LLMサービス | ドキュメント解析 |
| OCRサービス | 画像・スキャンPDFのテキスト化 |
| ファイルストレージ | 原本ファイルの保管 |
| メール通知サービス | リマインダー・アラート送信用 |

## 6. 業務フロー

### 6.1 全体像：出展社から主催者側への書類提出

```
【現状】
出展社 ──メール──► 装飾業者（コマ申込、設計書）
出展社 ──メール──► 電気施工会社（電気申込）
出展社 ──手渡し──► 弁当会社（弁当注文）
  ※ バラバラの形式・経路で送付、何をどこに送るかも曖昧

【本システム導入後】
出展社 ──Webアップロード──► 本システム
  │                           │
  │  ・ファイル選択            │ 提出カテゴリに応じて自動振り分け
  │  ・カメラ撮影              │
  │  ・提出カテゴリ選択         ├──► 装飾業者（コマ申込・ブース設営）
  │    （何の書類か選ぶ）       ├──► 電気施工会社（電気申込）
  │                           ├──► 弁当会社（弁当注文）
  │                           └──► 備品会社（備品申込）
  │
  └── 全てブラウザから操作      主催者は全体を統合管理

  ※ 提出カテゴリと受取先の組み合わせは展示会ごとに設定
```

### 6.2 メインフロー：ドキュメント提出〜データ確定

```
出展社              システム              主催者側
  │                   │                    │
  │ ① ポータルにログイン                    │
  │ ② 提出カテゴリを選択                     │
  │   (コマ申込/電気申込/弁当注文等)          │
  │ ③ ファイルアップロード                   │
  │   またはカメラ撮影                       │
  │ ──────────►       │                    │
  │                   │                    │
  │              ④ ファイル受信              │
  │              ⑤ ウイルススキャン           │
  │              ⑥ 原本保管                  │
  │              ⑦ テキスト抽出              │
  │                (カメラ画像→傾き補正→OCR) │
  │              ⑧ AI構造化解析              │
  │              ⑨ 信頼度スコア算出           │
  │                   │                    │
  │                   ├─ 信頼度≥閾値        │
  │                   │   → 自動確定         │
  │                   │   → ダッシュボード反映│
  │                   │                    │
  │                   ├─ 信頼度<閾値        │
  │                   │   → レビュー依頼通知  │
  │                   │                    │
  │                   │          ⑩ レビュー画面で確認
  │                   │           (原本と解析結果を
  │                   │            並べて表示)
  │                   │                    │
  │                   │          ⑪ 承認/修正/却下
  │                   │  ◄──────────────── │
  │                   │                    │
  │              ⑫ 確定データをDB格納        │
  │              ⑬ ダッシュボード更新         │
```

### 6.3 カメラ撮影フロー（スマホ）

1. 出展社担当者がスマホでポータルにアクセス
2. 「カメラで撮影」ボタンをタップ
3. 紙の書類をカメラで撮影
4. プレビュー画面で確認
   （傾き・明るさの自動補正）
5. 必要に応じて複数ページを連続撮影
6. 提出カテゴリを選択して送信（受取先は自動決定）
7. AI解析が自動で開始

## 7. 制約条件

| # | 制約 | 理由 |
|---|------|------|
| 1 | 出展社に専用ツールの利用を強制しない | ビジネス上の制約。ただしアップロードはWebポータルに統一 |
| 2 | 日本語のドキュメントが主対象 | 国内展示会が対象 |
| 3 | AI解析は100%の精度を保証しない | 人間レビューとの併用で品質担保 |
| 4 | 個人情報の取り扱いに注意 | 出展社担当者の氏名・連絡先等が含まれる |
| 5 | FAXは対応外とする | デジタル化を推進。カメラ撮影で紙書類にも対応 |

## 10. リスクと対策

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| AI解析精度が想定を下回る | 高 | 中 | プロンプトチューニング、学習データの蓄積、信頼度閾値の調整 |
| 出展社がポータルを使いこなせない | 中 | 中 | 極限までシンプルなUI、カメラ撮影で紙書類にも対応 |
| カメラ撮影画像の品質が低い | 中 | 中 | 自動補正機能、撮影ガイド表示、低品質時の再撮影促し |
| AI APIの料金が想定以上 | 中 | 低 | コスト監視、キャッシュ活用、バッチ処理による効率化 |
| 個人情報漏洩 | 高 | 低 | 暗号化、アクセス制御、監査ログ、定期セキュリティ監査 |

## 12. 用語定義

| 用語 | 定義 |
|------|------|
| 主催者 | 展示会を主催する企業・団体 |
| 装飾業者 | 主催者から展示会の全体管理（ブース施工含む）を委託された企業 |
| 主催者側協力会社 | 電気施工会社・弁当会社等、主催者側で書類を受け取る企業群 |
| 出展社 | 展示会にブースを出展する企業（書類を送る側） |
| 提出カテゴリ | 出展社が提出する書類の種類（コマ申込・電気申込・弁当注文等）。展示会ごとに自由に設定 |
| 受取先組織 | 提出カテゴリに紐付く、書類を受け取る主催者側の組織 |
| 注文書 | 出展社がブース設備（パネル、照明、電気工事等）を注文する書類 |
| 設計書 | ブースの設計図面・仕様書 |
| AI解析 | LLM（大規模言語モデル）によるドキュメントの構造化データ抽出 |
| 信頼度スコア | AI解析結果の確からしさを示す数値（0.00〜1.00） |
| レビューキュー | 人間の確認が必要なAI解析結果の待ち行列 |
| カメラ撮影機能 | スマホ等のカメラで紙書類を撮影し直接アップロードする機能 |
