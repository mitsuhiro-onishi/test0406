# 展示会ドキュメント一元管理システム — システム全体設計書

## 1. プロジェクト概要

### 1.1 解決すべき課題

展示会の主催者・装飾業者・主催者側の協力会社（電気施工会社、弁当・ケータリング会社等）は、
出展各社から**様々な形式**（Excel, Word, PDF等）で注文書・設計書を受領している。

| 課題 | 詳細 |
|------|------|
| フォーマットの不統一 | Excel, Word, PDF, スマートフォン撮影画像など |
| 手作業での転記 | 受領→確認→手入力の繰り返しでミス・工数が発生 |
| 出展社への負担 | 主催者独自ツールの強制は現場の反発を招く |
| 紙書類のデジタル化 | カメラ撮影画像の歪み・明るさのばらつきへの対応 |

### 1.2 ソリューションコンセプト

> **「出展社は今まで通りのドキュメントを送るだけ。AIが中身を読み取り、主催者のDBに自動統合する」**

```
出展社A ──┐  Excel
出展社B ──┤  Word     ┌──────────────┐    ┌─────────────┐    ┌──────────────┐
出展社C ──┼─ PDF ───► │ 提出カテゴリ  │──►│  AI解析基盤  │──►│  統合DB      │──► 管理画面
出展社D ──┤  カメラ撮影 │  選択        │    │  (OCR+LLM)  │    │  (構造化)    │
出展社E ──┘  画像     └──────────────┘    └─────────────┘    └──────────────┘
```

### 2.2 技術スタック（推奨）

| レイヤー | 技術 | 理由 |
|---------|------|------|
| フロントエンド | Next.js (React) + TypeScript | SSR対応、管理画面に最適 |
| API | Node.js (NestJS) or Python (FastAPI) | AI処理はPython、APIはどちらでも |
| AI解析 | Claude API (Anthropic) | ドキュメント理解に優れたLLM |
| OCR | Google Cloud Vision API / Tesseract | 画像・スキャンPDFのテキスト化 |
| 画像補正 | OpenCV / Pillow | カメラ撮影画像の傾き補正・明るさ調整 |
| HEIC変換 | pillow-heif / libheif | iPhoneカメラ撮影画像(HEIC)のJPEG変換 |
| DB | PostgreSQL | リレーショナルデータ管理 |
| ファイル保管 | AWS S3 / MinIO | 原本の長期保管 |
| 全文検索 | Elasticsearch | 過去ドキュメントの横断検索 |
| 認証 | Auth0 / Keycloak | マルチテナント対応 |
| インフラ | AWS / GCP | スケーラビリティ |

## 5. AI解析パイプライン詳細

### 5.1 処理フロー

```
ファイル受信（Webアップロード / カメラ撮影）
    │
    ▼
┌──────────────┐
│ ① 前処理     │  ・MIMEタイプ判定
│              │  ・ウイルススキャン
│              │  ・ファイル正規化
│              │  ・HEIC→JPEG変換
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ ② 画像補正   │  ・傾き補正（台形歪み補正）
│              │  ・明るさ・コントラスト自動調整
│              │  ・ノイズ除去
│              │  ※カメラ撮影画像の場合のみ実行
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ ③ テキスト   │  Excel → openpyxl でセル値抽出
│    抽出      │  Word  → python-docx で段落抽出
│              │  PDF   → pdfplumber / PyMuPDF
│              │  画像  → OCR (Cloud Vision / Tesseract)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ ④ LLM解析    │  Claude API に以下を送信:
│              │  ・抽出テキスト
│              │  ・ドキュメント種別ヒント
│              │  ・構造化抽出プロンプト
│              │
│              │  出力: JSON (構造化データ)
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ ⑤ 検証       │  ・必須フィールドチェック
│              │  ・データ型・範囲チェック
│              │  ・信頼度スコア算出
│              │  ・過去データとの整合性チェック
└──────┬───────┘
       │
       ├── 信頼度 ≥ 閾値 ──► DB格納（自動承認）
       │
       └── 信頼度 < 閾値 ──► 人間レビューキュー

```

### 5.2 対応ファイル形式

| 形式 | 拡張子 | 備考 |
|------|--------|------|
| Excel | .xlsx, .xls | openpyxl で解析 |
| Word | .docx, .doc | python-docx で解析 |
| PDF | .pdf | テキストPDF / スキャンPDF両対応 |
| JPEG | .jpg, .jpeg | カメラ撮影画像含む |
| PNG | .png | スクリーンショット等 |
| HEIC | .heic, .heif | iPhone撮影画像、JPEG自動変換後に処理 |

### 5.3 LLM構造化抽出プロンプト例

```
以下のドキュメントは展示会「{exhibition_name}」の
出展社「{company_name}」から送付された{document_type}です。

以下の情報を JSON形式で抽出してください:

【注文書の場合】
- order_items: 注文品目リスト
  - item_name: 品目名
  - quantity: 数量
  - unit: 単位
  - unit_price: 単価
  - total_price: 合計金額
- delivery_date: 希望納期
- booth_number: ブース番号
- special_instructions: 特記事項

【設計書の場合】
- booth_number: ブース番号
- dimensions: 寸法 (幅 x 奥行 x 高さ)
- materials: 使用素材リスト
- electrical: 電気工事要件
- special_requirements: 特殊要件

--- ドキュメント内容 ---
{extracted_text}
```

## 6. セキュリティ要件

| 項目 | 対策 |
|------|------|
| 認証 | OAuth 2.0 / OIDC (Auth0 / Keycloak) |
| 認可 | RBAC（ロールベースアクセス制御） |
| 通信暗号化 | TLS 1.3 必須 |
| データ暗号化 | AES-256 (保存時暗号化) |
| ファイルスキャン | ClamAV によるウイルススキャン |
| 監査ログ | 全操作の監査ログ記録 |
| マルチテナント | テナント間データ完全分離 |
| GDPR/個人情報 | 個人情報マスキング、保持期間管理 |
